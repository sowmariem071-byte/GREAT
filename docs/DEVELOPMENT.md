# 内容排期系统开发协作规范

本文档记录项目开发时必须遵守的协作规则。业务需求以 [开发需求文档](./REQUIREMENTS.md) 为主线，执行顺序以 [开发计划文档](./PLAN.md) 为准，真实变更记录写入 [开发进度文档](./PROGRESS.md)。

## 1. 文档体系

- `docs/REQUIREMENTS.md`：需求主线，记录已经确认的业务规则、页面、角色、权限、视觉原则和验收标准。
- `docs/PLAN.md`：开发计划，记录阶段状态、下一步优先级、页面级执行顺序和验证要求。
- `docs/PROGRESS.md`：开发进度，记录每次变更的目标、内容、影响文件、验证和下一步。
- `docs/THEME_CUSTOMIZATION.md`：配色方案模块设计。
- `docs/README.md`：文档入口和同步规则。

## 2. 开发顺序

每次开发遵守以下顺序：

1. 明确本次任务是否改变需求主线。
2. 需求变化先更新 `docs/REQUIREMENTS.md`。
3. 执行顺序或优先级变化时更新 `docs/PLAN.md`。
4. 修改代码、样式、数据结构或文档。
5. 根据变更范围执行验证。
6. 追加 `docs/PROGRESS.md`。

## 3. 当前项目形态

项目已经从静态 UI 原型升级为 Next.js 全栈应用。

正式项目主要文件：

- `src/app`：Next.js App Router 页面、认证接口和服务端动作。
- `src/components`：应用壳层、弹窗、统计卡片、状态标签、脚本详情、视频详情等复用组件。
- `src/lib`：认证、Prisma、时间、标签、自动分配、提醒和对象存储工具。
- `prisma/schema.prisma`：数据库模型。
- `prisma/migrations/20260706143000_init/migration.sql`：初始 PostgreSQL 迁移。
- `prisma/seed.ts`：初始用户和目标种子数据；不再写入测试脚本、视频、审核、排期或通知。
- `Dockerfile`、`docker-compose.yml`：Docker 部署配置。
- `.env.example`：环境变量示例。
- `scripts/setup-wsl-docker.ps1`：Windows 本机启用 Docker Desktop 所需 WSL2 功能的辅助脚本。

静态原型仍作为视觉参考保留：

- `index.html`
- `styles.css`
- `app.js`
- `preview-*.png`

## 4. 业务开发约定

- 三类角色固定为管理员、编导、剪辑。
- 数据权限必须按角色控制。
- 编导和剪辑只能处理自己权限范围内的数据。
- 管理员可以查看和管理全局数据。
- 需要填写的新增、下达、审核、排期、发布、人员目标和设置操作，优先使用弹窗。
- 审核修改必须保留历史轮次。
- 视频发布第一版只做手动标记，不对接外部平台。
- 视频成片第一版只保存链接，不上传成片文件。
- 头像和审核截图走 S3 兼容对象存储。
- 时区固定为 Asia/Shanghai。

## 5. UI 开发约定

- 主视觉基调为暖灰白背景、白色或半透明玻璃卡片、深灰黑主文字、浅灰辅助文字、赤陶红或焦糖橙点缀。
- 页面必须有留白，避免信息堆满。
- 首页只放与岗位工作进度强相关的信息。
- 数据卡片需要有层级、呼吸感和轻阴影，避免普通后台表格感。
- 弹窗需要轻量动效，打开和关闭都不能突兀。
- 移动端和桌面端都不能出现明显横向溢出。

## 6. 验证要求

代码和样式变更优先执行：

```powershell
npx prisma validate
npm run typecheck
npm run lint
npm run build
```

纯文档变更至少执行：

```powershell
rg -n "开发需求文档|开发计划文档|开发进度文档" docs
```

涉及页面交互时，还需要通过浏览器验证关键路径，例如登录、打开弹窗、提交表单、点击状态入口和跳转目标页面。

## 7. 进度记录格式

`docs/PROGRESS.md` 使用以下格式追加记录：

```md
## YYYY-MM-DD HH:mm - 标题

- 目标：
- 变更：
- 影响文件：
- 验证：
- 下一步：
```

每次记录必须写清楚本次变更解决了什么、改了哪里、验证了什么，以及后续应该继续做什么。
