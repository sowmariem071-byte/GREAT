# 配色方案模块设计

## 目标

让使用人可以在系统里调整自己的界面配色，同时不破坏整体高级、轻盈、浅色玻璃拟态的视觉基调。

## 使用范围

- 每个用户可以设置自己的个人配色。
- 管理员可以设置团队默认配色。
- 新用户默认使用系统推荐的浅蓝薄荷主题。
- 配色只影响个人界面观感，不影响业务数据和其他人的界面。

## MVP 功能

第一阶段建议实现以下能力：

- 预设主题选择：浅蓝薄荷、粉紫创作、青绿剪辑、低饱和灰蓝。
- 自定义点缀色：主色、辅助色、强调色。
- 实时预览：设置面板内显示数据卡、按钮、统计环和进度条的预览。
- 一键恢复默认。
- 保存到本地或用户配置。

## 当前实现状态

静态原型已实现第一阶段配色功能：

- 左侧“设置”按钮可打开配色调整面板。
- 支持 4 个预设主题。
- 支持主色、辅助色、强调色 3 个颜色选择器。
- 选择预设或修改颜色后会实时预览。
- 点击保存后写入 `localStorage`。
- 刷新页面后自动应用已保存配色。
- 点击恢复默认后清空自定义配色，回到当前角色默认主题。

正式 Next.js 项目已实现用户级配色保存：

- 数据表为 `ThemePreference`。
- 设置页为 `/settings`。
- 支持 4 个预设主题和 3 个自定义颜色。
- 保存后写入数据库，并在 `AppShell` 中以 CSS 变量覆盖角色默认主题。
- 当前优先级为：用户自定义主题 > 角色默认主题。

## 当前可复用的主题变量

当前 `styles.css` 已经使用以下 CSS 变量控制主要配色：

```css
--accent
--accent-2
--accent-3
```

当前角色主题定义为：

```css
[data-theme="admin"]
[data-theme="director"]
[data-theme="editor"]
```

后续可扩展为：

```css
[data-user-theme="mint"]
[data-user-theme="pink"]
[data-user-theme="cyan"]
[data-user-theme="custom"]
```

## 建议数据结构

静态原型阶段可先使用 `localStorage`：

```json
{
  "themePreset": "mint",
  "accent": "#64cfee",
  "accent2": "#83e6cd",
  "accent3": "#9f91ff"
}
```

正式系统阶段建议存入用户配置表：

```text
user_theme_preferences
- id
- user_id
- preset_name
- accent
- accent_2
- accent_3
- background_style
- updated_at
```

## 交互方案

设置入口放在左侧导航底部的“设置”中。

设置面板包含：

- 当前主题名称。
- 4 个预设主题卡片。
- 3 个颜色选择器：主色、辅助色、强调色。
- 预览区域：按钮、数据卡、进度条、统计环。
- 操作按钮：保存、恢复默认、取消。

保存逻辑：

1. 用户选择预设主题或修改颜色。
2. 页面立即预览新颜色。
3. 点击保存后写入配置。
4. 下次进入系统时自动读取并应用。

## 校验规则

- 自定义颜色必须是合法 HEX 色值。
- 主色、辅助色、强调色不能全部过浅，避免按钮和图表不可见。
- 文本颜色不跟随用户自由修改，保持深灰蓝，保证可读性。
- 背景仍保持浅色渐变，不开放纯黑、纯红等破坏系统气质的主题。

## 与角色主题的关系

默认情况下，角色主题提供轻微差异：

- 管理员：蓝、薄荷、紫。
- 编导：蓝、绿、粉紫。
- 剪辑：蓝、青绿、紫。

用户自定义主题优先级更高。实现时建议优先级为：

```text
用户自定义主题 > 用户选择的预设主题 > 角色默认主题 > 系统默认主题
```

## 实现步骤建议

1. 把主题变量从角色数据中抽象为统一主题配置。
2. 增加设置面板 UI。
3. 增加主题预览和应用函数。
4. 使用 `localStorage` 保存静态原型的用户选择。
5. 正式接后端后迁移到用户配置表。
6. 在 `docs/PROGRESS.md` 记录每次配色模块改动。

## 验收标准

- 用户可以选择预设主题并立即看到界面变化。
- 用户可以自定义三种点缀色。
- 刷新页面后仍保留用户选择。
- 恢复默认后回到角色默认主题。
- 三类主页在自定义配色下仍保持文字清晰、布局稳定。
